ADVISING EVIDENCE — ADV2
Compliance Strategy Recommendation
Marketing Agent POC

Date: after ADV1 completed, before guardrails.py was written
Audience: self
Purpose: resolve the compliance mechanism choice before writing any code,
since getting it wrong means rebuilding the most legally consequential agent.


Recommendation
--------------
Use pure Python regex for compliance checking. Do NOT use an LLM judge.
Do NOT use an external moderation API.


Why this, specifically
-----------------------

1. The failure mode of an LLM judge is a false negative, not a false positive

   An LLM that decides context makes a guarantee claim acceptable is making
   a legal judgment call. "Results guaranteed or your money back" — an LLM
   will often consider the refund offer a mitigating factor and not flag it.
   That is an FTC violation in real ad copy.

   False positives (flagging clean copy) are annoying. False negatives (missing
   a guarantee claim) are liability. The risk profile is asymmetric.

2. "Guaranteed" is always "guaranteed" — there is no context that makes it acceptable

   Compliance is a lookup problem, not a reasoning problem. The seven required
   claim categories (FR7) are strings. They either appear in the copy or they
   don't. An LLM adds probabilistic reasoning to a deterministic problem.
   Regex matches or it doesn't.

3. Runtime matters for a five-agent sequential pipeline

   An LLM compliance judge adds 10-30 seconds to the pipeline. Pure Python
   regex adds less than 1 millisecond. In a pipeline that already takes 60-90
   seconds, that difference is meaningful.

4. Auditability is required

   A compliance decision needs to be explainable: "this flag appeared because
   the word 'guaranteed' matched pattern X, and the reason is Y." An LLM
   judge produces reasoning that sounds confident but is not traceable to a
   specific rule. The regex approach produces: matched text, variant ID, reason
   string — three fields, fully auditable.


What to build, concretely
--------------------------

  agents/guardrails.py:

    FLAGGED_PATTERNS = [
        (regex, "human-readable reason string"),
        ...  # seven patterns for seven FR7 categories
    ]

    def check(copy: dict) -> dict:
        # iterate FLAGGED_PATTERNS against variant_1 and variant_2
        # return { clean_copy: {variant_1, variant_2}, flags: [...] }

  Each flag entry: { variant: "variant_1", matched: "guaranteed", reason: "..." }

  The function returns clean_copy (unchanged — no text modification) and flags
  (list of matches). The UI renders flags as warnings; the user decides whether
  to edit.

  guardrails.py must have NO LLM call, NO network call, NO external dependency.
  It must be testable with `python -c "from agents import guardrails; ..."` with
  no server running.


Seven required patterns (FR7)
------------------------------
  1. guarantee_claim    — "guaranteed", "guarantee"
  2. health_claim       — "cured", "heals", "treats"
  3. superlative_claim  — "#1", "number one", "best in the world"
  4. free_offer         — "free trial", "no risk"
  5. urgency_language   — "limited time", "act now", "last chance"
  6. scientific_claim   — "scientifically proven", "clinically tested"
  7. weight_loss_claim  — "lose weight", "weight loss"

  All patterns case-insensitive. Each has a one-line reason string for display.


Coverage gap to document
-------------------------
Novel phrasing will slip through. "Tested in partnership with leading
dermatologists" does not match the scientific_claim pattern. "Results will
follow" does not match guarantee_claim. The coverage is necessary but not
exhaustive. Document this gap in the DL so users know human review is still
required before publishing.


Risks
-----
- Over-flagging urgency language is likely. "Shop now" could be read as urgency.
  Pattern must be specific enough to avoid flagging standard CTAs.
- As the copywriter prompt evolves, new violation patterns may emerge that
  are not in the initial seven. The pattern list is a maintenance surface.
