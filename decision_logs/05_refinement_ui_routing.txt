DL-05 — Refinement UI and ReAct routing
Marketing Agent POC


0. Context

The first result page was one-shot. Read the output, want a different tone,
go back, re-upload, wait 90 seconds. That's not a usable workflow. This DL
covers the UI design and the routing logic that replaced it.

Also: the CTA optimizer was returning "Variant 1" instead of "variant_1" and
breaking the template with a 500. Fixed in this sprint.


1. Research question

How do you let a user iterate on campaign output without re-running the full
pipeline — and how do you make the agent routing visible rather than a black box?


2. Current LO stage

3. What makes a good decision here

- Vision never re-runs during refinement (product didn't change)
- Routing decisions are visible — user sees which agents ran
- Sliders are instant; LLM calls only happen on "Refine"
- The system surfaces suggestions before the user has to ask


4. What I decided

Two-column sticky layout (wireframe.txt). Sliders above chat. Routing label
visible. Proactive /suggest call on page load. ReAct single-pass routing: one
LLM call decides which agents re-run, then runs them in pipeline order.

For generation: SSE streaming so the user sees each agent step as it happens
rather than staring at a blank screen for 90 seconds.


5. Why this decision

Method: Lab + Showroom

The wireframe drove the layout. Two-column beats tabbed or stacked because
report and refinement are used simultaneously — tabbed means context switching,
stacked means scrolling past the full report to reach the panel.

Sliders sit above chat because slider changes are deterministic (no LLM call
until Refine is clicked). The fast path should be the first thing a user encounters.

The routing label was a direct requirement from the assignment: "visible agent
collaboration." Users must see which agents ran, not just that output changed.

Proactive suggestions: the assistant analyses the output on page load and posts
2-3 specific improvements before the user has typed anything. In 3 tests, it
surfaced actionable suggestions on 2/3 campaigns. The 1 generic response was on
a campaign that was already high-quality — not a failure.

Stateless /refine: full output state travels as JSON in every request. I'd
planned server-side session state initially. The stateless version turned out
simpler and means the tool deploys anywhere without session infrastructure.

The CTA key bug: mistral:7b returned "Variant 1" (capitalised, space) instead
of "variant_1". Jinja2 threw KeyError. Fix: normalise all LLM output keys to
snake_case before returning from any agent function.

What I'd recommend:
Design the wireframe before building the template. I avoided three layout
mistakes (tabbed panel, export at top, routing hidden) because they were
visible in the ASCII spec before any HTML was written.

GAP found:
Refinement history is client-side only. If the user refreshes, the chat and
routing history disappear. The outputState persists in the JS variable but the
conversation context is lost.


6. Does this hold up

- Vision skipped in /refine:        ✅ vision/color never appear in agents_to_run
- Routing label visible:            ✅ routing_label rendered in right panel after each call
- Proactive suggestions fire:       ✅ /suggest called on DOMContentLoaded
- SSE progress steps show:          ✅ vision → audience → copy → guardrails → CTA
                                        labels update as each agent finishes
- CTA keys consistent:              ✅ variant_1/variant_2 normalised; no 500s since fix

Assumptions I'm making:
Browser supports Fetch with ReadableStream (all modern browsers do). No IE11.

What surprised me:
The stateless design was more robust than the session approach I'd planned.
Every /refine request is independently auditable — full input and output are
in the request/response logs. Nothing hidden in server state.


7. What this unlocks

Next LO stage: Realizing → Managing (pipeline is now a working interactive tool,
not just a one-shot report generator)

What I can now do:
Demo the full iteration loop: generate → see routing → adjust tone slider →
refine → see updated copy — all in one session without re-uploading.

Ongoing monitoring:
Watch routing_label in the chat history. If it's always "Copywriter → CTA →
Output" regardless of what the user asked, the routing LLM is defaulting too
aggressively — prompt needs tuning.

How I'll know this worked:
"Make the CTA more urgent" routes to cta_optimizer only. "Change the tone to
playful" routes to copywriter + guardrails + cta_optimizer. Vision never appears
in any routing label during refinement.
