DL-07 — Model Split: Creative Copy vs Structured JSON
Marketing Agent POC


0. Context

The pipeline had run on mistral:7b for every text task since DL-01. The pipeline design had named the risk early: mistral:7b might time out on long-context copywriting (RAG chunks plus persona plus description), so keep a fallback creative model in mind¹. The timeout arrived on a run with a longer description and four RAG chunks — exactly the predicted scenario. Fixing it confirmed an improvement that had been available from the start.

Keeping the whole pass under a minute is the TO-BE picture's speed claim — days of agency relay collapsed into about a minute of machine work (@gap-to-be). This decision is what keeps that claim true under load, and it feeds DL-08, where pipeline_s across runs becomes an early confirming finding.


1. Research question

Is one general-purpose model right for all text tasks, or do creative and structured-output tasks benefit from different models?

3. What makes a good decision here

- The copywriter completes in under 60 seconds on a mid-range laptop
- The JSON agents (audience, CTA, routing) stay reliable
- A config change is one line per model — no code change to swap


4. What I decided

gemma4:e2b for the copywriter only. mistral:7b for everything else. The split lives in config as OLLAMA_MODEL and OLLAMA_FAST_MODEL, and each agent imports the constant it needs.


5. Why this decision

Method: Lab.

mistral:7b timed out at 60 seconds on a roughly 3,500-token context — the description plus four RAG chunks. The upper bound is 120 seconds for the whole campaign (@constraints), and the copywriter alone eating the budget before CTA optimisation was the concrete problem.

Three replacements tested as copywriter alternatives:

  llama3.2:1b   — 8s, both variants nearly identical and very generic
  gemma4:e2b    — 25-40s, noticeably better variety and emotional range
  gemma4:4b     — 45-90s, similar quality but the timeout risk returns

gemma4:e2b on the input that had timed out on mistral produced "Stay hydrated, stay green. 24h cold, zero plastic compromise. #LiveGreen", versus mistral's truncated "Stay hydrated with our eco-friendly bottle. Shop now." The variant-angle constraint — one emotional hook, one product benefit — needs a model with enough range to produce two structurally distinct outputs consistently (@agent-contracts). At 1B parameters, llama could not; gemma4:e2b could.

[IMAGE: diagrams/pipeline-dataflow.png — gemma4 drives the copywriter; mistral drives the structured agents around it]

mistral:7b stays on the JSON tasks — audience, CTA scoring, and routing — because its structured-output reliability is measurably better. In testing, mistral returned valid JSON on 19 of 20 schema-constrained calls; gemma4 on about 14 of 20. The audience, CTA, and routing functions all depend on the model returning the right schema (@agent-contracts), so a 30% parse-failure rate is a real cost. The principle is the same one the pipeline rests on — each agent gets only what it needs — extended to model choice: the copywriter needs expressive range, the router needs schema reliability.

What I'd recommend:
Measure before assuming one model handles every task. The risk was named early; the timeout made it concrete. Without the incident, one model for everything would have looked fine until a long-context demo exposed it.

GAP found:
No controlled comparison of gemma4:e2b vs mistral:7b on the same copywriting prompt. The switch was justified by the timeout and side-by-side reads, not a formal evaluation (@research-questions).


6. Does this hold up

- Copywriter under 60s:   pass — gemma4:e2b averages 25-40s in production
- JSON reliability:       pass — mistral:7b at 19/20 valid JSON on the test set
- One-line config swap:   pass — OLLAMA_MODEL / OLLAMA_FAST_MODEL in config
- Pipeline under 120s:    pass — measured 70-95s after the split

Assumptions I'm making:
gemma4:e2b is pulled before the server starts. setup.py handles this; if the model is missing, the copywriter returns an error dict rather than crashing.

What surprised me:
The risk was named, the timeout was the predicted scenario, and the fix confirmed an improvement that should have been in the original design. Acting on it earlier would have been better.


7. What this unlocks

Next LO stage: Managing (the pipeline is stable; focus shifts to
observability).

What I can now do:
Swap either model independently. Better copy means changing one constant; better routing means changing the other. They do not interfere.

Ongoing monitoring:
pipeline_s in run_log.jsonl. If copywriter time rises above 50 seconds, either context is growing or Ollama is under memory pressure — the split lets either be diagnosed alone.

How I'll know this worked:
No copywriter timeouts across 10 consecutive runs. pipeline_s stays under 100 seconds even with 4 RAG chunks in context.
