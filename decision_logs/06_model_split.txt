DL-06 — Model timeout forces split: gemma4:e2b for copy, mistral:7b for JSON
Marketing Agent POC


0. Context

The pipeline was running on mistral:7b for everything. Then the copywriter timed
out at 60 seconds during a run with a long description and four RAG chunks. The
fix revealed a design improvement that was available from the start.


1. Research question

Is one general-purpose model the right choice for all text tasks, or do creative
and structured-output tasks benefit from different models?


2. Current LO stage

[ ] Analyzing
[ ] Advising
[ ] Designing
[x] Realizing  — timeout triggered model comparison; tested 3 alternatives
[x] Managing   — split wired in config; model_used logged per run; startup check added


3. What makes a good decision here

- Copywriter completes in <60 seconds on a mid-range laptop
- JSON-structured agents (audience, CTA, routing) stay reliable
- Config change is one line per model — no code changes needed to swap


4. What I decided

gemma4:e2b for the copywriter only. mistral:7b for everything else.
Split lives in config.example.py as OLLAMA_MODEL and OLLAMA_FAST_MODEL.
Each agent imports the model constant it needs.


5. Why this decision

Method: Lab

mistral:7b timed out at 60s with a ~3,500 token context window (description +
4 RAG chunks). It appeared to stall generating the second variant.

Three models tested as copywriter replacements:
  llama3.2:1b — 8s but variants were nearly identical, very generic
  gemma4:e2b  — 25-40s, noticeably better syntax variety and emotional range
  gemma4:4b   — 45-90s, similar quality to e2b but timeout risk with long context

gemma4:e2b output on the same input that timed out on mistral:

  "Stay hydrated, stay green. 24h cold, zero plastic compromise. #LiveGreen"
  vs mistral's pre-timeout: "Stay hydrated with our eco-friendly bottle. Shop now."

mistral:7b stays for JSON tasks (audience, CTA, routing) because its structured
output reliability is better than gemma4 — 19/20 valid JSON returns vs ~14/20
for gemma4 on schema-constrained prompts.

The insight: model size should match task complexity, not task importance.
Creative prose with open-ended generation benefits from a model with more
expressive range. JSON extraction of a fixed schema is simpler — a faster,
more reliable model wins there.

What I'd recommend:
Measure this before assuming one model fits all tasks. The timeout was the
signal. Without it, I might have run mistral:7b for everything and just assumed
the copywriter was "good enough."

GAP found:
No systematic comparison of gemma4:e2b vs mistral:7b output quality on the
same copywriting prompt. The switch was justified by the timeout and a few
side-by-side reads, not a controlled test.


6. Does this hold up

- Copywriter <60s:            ✅ gemma4:e2b averages 25-40s in production
- JSON reliability:           ✅ mistral:7b at 19/20 valid JSON on test set
- One-line config swap:       ✅ OLLAMA_MODEL / OLLAMA_FAST_MODEL in config.py
- Pipeline <120s total:       ✅ measured at 70-95s after split

Assumptions I'm making:
gemma4:e2b is pulled before the server starts. scripts/setup.py handles this.
If it's missing, the copywriter falls back to an error dict — no crash,
but no copy either.

What surprised me:
The timeout event exposed something useful: the pipeline assumed uniform model
performance across different task types. That assumption was wrong. The split
is better architecture regardless of the timeout.


7. What this unlocks

Next LO stage: Managing (pipeline is stable; focus shifts to observability)

What I can now do:
Swap either model independently. Upgrading copy quality means changing one
config line. Improving routing reliability means changing a different one.

Ongoing monitoring:
pipeline_s in run_log.jsonl. If copywriter starts creeping above 50s,
either context is growing or Ollama is under memory pressure.

How I'll know this worked:
No copywriter timeouts across 10 consecutive runs. pipeline_s stays under 100s
even with 4 RAG chunks in context.
