DL-03 — Vision Model and Output Schema: What the Pipeline Sees First
Marketing Agent POC


0. Context

Vision is the first agent, and everything downstream depends on what it returns. The pipeline design had flagged the risk before the first run: a vision model must be vision-capable (the ":vl" tag), or it returns a 400 error — verify before picking a name¹. I did not verify. The first run failed exactly that way, and fixing it taught me that the output schema mattered far more than which model ran.

Vision is the product-analysis function from the AS-IS agency, now the front of the pipeline (@gap-analysis). Its target_signals field becomes the primary context the audience agent builds on (@agent-contracts), which is what makes the persona jump in DL-02 possible.


1. Research question

Which vision model works reliably on the OpenRouter free tier, and what output schema makes audience segmentation as effective as possible?

3. What makes a good decision here

- The vision agent returns valid JSON every time, or fails gracefully
- The output carries buyer signals, not just visual descriptions
- The fallback activates automatically, without the user seeing an error


4. What I decided

A two-model fallback chain: nemotron-nano-12b-v2-vl:free primary,
llama-3.2-11b-vision-instruct:free fallback. Redesign the output schema from visual attributes to buyer signals — target_signals is the key field, not colours or materials.


5. Why this decision

Method: Lab.

The first model choice was a "-it" (instruction-tuned) model and returned an immediate 400: "model does not support vision." The pipeline design had written that exact warning¹. Switching to the ":vl" (vision-language) model worked on the first try.

The schema redesign came from reading what the audience agent actually takes: it receives the full vision output (product_type, product_tags, mood, target_signals), not an image and not raw descriptions (@agent-contracts). The initial schema returned the wrong things:

  Before: visual_style, colors, materials, overall_impression
  After:  product_type, product_tags, mood, target_signals, suggested_hashtags

The audience agent had been receiving "matte green finish, stainless steel" and trying to infer a buyer from material words. target_signals — "eco-conscious", "fitness enthusiast", "outdoor lifestyle" — hands it the product already in buyer-identity language. Before and after on the same bottle:

  Audience before redesign: "Health and Wellness Consumer, 20-45"
  Audience after redesign:  "Eco-Conscious Millennial Athlete, 25-34"

Same model downstream. Different input quality.

[IMAGE: diagrams/pipeline-dataflow.png — vision's target_signals flowing into the audience agent as its primary context]

The fallback chain was added after confirming free-tier models go offline without notice — a risk the pipeline design named: a single-model vision stage blocks the whole pipeline, so vision needs a fallback¹. In 10 test runs, 2 fell back to llama; both finished cleanly. If both fail, vision returns a safe dict with empty tags so downstream agents run rather than crash, though the output will be generic. That degradation is logged but not yet surfaced in the UI.

What I'd recommend:
Check the modality tag before picking any model for a vision task — names do not imply modality. Then test the output schema against the downstream agent's real needs before calling the design done. The schema decision had more impact than the model choice.

GAP found:
If both free-tier vision models go offline at once, the pipeline degrades silently. A UI warning when the fallback fires would tell the user why the output came back generic. The spec does not cover this — it is an operational gap that emerged in practice.


6. Does this hold up

- Returns valid JSON:      pass — nemotron consistent across 10 test runs
- Fallback activates:      pass — a forced nemotron failure served the llama fallback cleanly
- Buyer signals in output: pass — target_signals is the field the audience agent uses first

Assumptions I'm making:
The OpenRouter free tier stays available during a demo. 2 of 10 runs fell back; both completed. The fallback chain works as designed.

What surprised me:
The schema redesign mattered more than the model choice. nemotron versus llama made little difference; buyer signals versus visual attributes made a large one.

How I'll know this worked:
