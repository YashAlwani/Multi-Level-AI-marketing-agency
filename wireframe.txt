DESIGN DECISIONS CAPTURED IN THIS WIREFRAME
────────────────────────────────────────────

1. Right panel is STICKY (position: fixed equivalent)
   Rationale: campaign report is long (scroll). Refinement should always be
   accessible without scrolling back to top. User needs to read copy AND
   refine in the same view.

2. Sliders sit ABOVE the chat input, not below
   Rationale: slider changes are deterministic (no LLM call). They should
   be the first thing a user tries — cheap, instant, reversible.
   Chat is for freeform requests that need LLM reasoning (slower).

3. Routing label is VISIBLE ("Routing: CTA → Output")
   Rationale: the assignment requirement is "visible agent collaboration."
   Users must see which agents are running on each refine, not just see
   the output change. This makes the multi-agent structure legible.

4. Proactive suggestion appears in chat ON PAGE LOAD (not after user asks)
   Rationale: the assistant analyses the output immediately and surfaces
   2-3 improvement suggestions. This mirrors how a real account manager
   would review creative before the client does. The user doesn't have to
   know what to ask — the system tells them what to look at first.

5. Two-column layout (not tabbed, not stacked)
   Rationale: report and refinement are in use simultaneously. Tabbed
   layout would require switching back and forth. Stacked layout means
   scrolling past the entire report to reach the refine panel. Side-by-side
   keeps both in view — same layout pattern as code editors with preview.

6. Variant label annotation ("emotional hook" / "product benefit")
   Rationale: the copywriter prompt forces two distinct angles. The UI
   labels them so the user understands WHY the variants differ — not just
   that they do. Transparency about the generation constraint is useful
   when deciding which to iterate on.

7. Character count shown per variant (143c, 128c)
   Rationale: FR4 sets a 125-char limit for primary text. Users need to
   know if they're over the limit after refinement changes the copy.
   Counts update after each /refine response.

8. Export button placed at BOTTOM of report (not top)
   Rationale: user reads the full report before deciding to export.
   Top placement would suggest "download before reviewing." Bottom
   placement is the natural end of the reading flow.

INTERACTION STATES
──────────────────
State 1 — Page load:
  Refinement panel shows sliders at current values.
  Chat shows proactive suggestion from /suggest (fires immediately).

State 2 — User moves slider:
  Slider value updates live (JS).
  No server call until "Refine Campaign" is clicked.

State 3 — Refine in progress:
  Refine button disabled.
  Routing label shows "Running: [agents]..."
  Chat shows "Agent is working..." indicator.

State 4 — Refine complete:
  Only affected sections of the report update in-place (DOM patch).
  Routing label shows what ran.
  Chat shows agent response summarising what changed.
  Button re-enables.

State 5 — Error:
  Chat shows error message.
  Report sections unchanged.
  Button re-enables.
